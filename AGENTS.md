# Stock 项目记忆

> 最后更新：2026-07-12

## 项目目标

本项目是面向用户个人的 A 股供应链错配研究系统。每日从免费公开来源采集新闻、公告、期货和行情数据，识别供需错配，沿供应链知识图谱映射行业、ETF 和股票，结合用户持仓生成 1～4 周投资研究建议，并通过飞书私聊推送。

系统只提供研究建议，不自动下单。所有结论必须附证据、时间、来源、置信度、风险和失效条件。

## 用户与投资约束

- 资金规模：约 50 万元。
- 可接受组合最大回撤：20%。
- 主要周期：1～4 周产业趋势，兼顾短线催化。
- 市场范围：第一期为 A 股和行业 ETF；期货仅作为供应链价格证据。
- 输出深度：股票评分、观察/介入区间、止损参考、失效条件和持仓动作建议。
- 数据预算：仅使用免费公开来源。
- LLM：Dify 中的 MiniMax M2.7 Highspeed。
- 推送对象：仅用户本人，复用当前项目已配置的飞书 App 身份。
- 默认推送：工作日 08:20 早报，高置信突发事件即时提醒，20:30 简短复盘。

## 架构决策

Dify 是用户可见的主控制台，不只是报告生成器。定时触发、流程分支、质量门禁、MiniMax 分析、持仓诊断、报告生成、飞书推送和运行状态必须在 Dify 工作流中清晰可见。

Dify 是生产环境唯一主调度器。现有 Python APScheduler 默认关闭，仅作为人工启用的灾备方案；两者禁止同时调度同一生产任务，避免重复采集、数据竞争和重复推送。

Python 服务负责 Dify 不适合承担的确定性和长耗时能力：多源采集、正文抽取、去重、数据库、行情指标、知识图谱传播、回测和评分基础数据。所有能力通过幂等 HTTP API 暴露给 Dify，每次调用返回 run_id、数据量、质量状态、耗时和错误摘要。

Dify Code 节点禁止直接做网络采集或复杂计算；本机 Dify Code 节点不能使用 requests、不能调用 MCP，且有约 30 秒限制。

## 不可妥协的质量规则

- “任务执行成功”和“数据质量合格”必须是两个状态。
- 数据缺失、过期、单一来源或相互矛盾时，不得生成强推荐。
- 新闻至少做标题/正文相关性、发布时间、来源可信度、重复事件聚类和反向证据检查。
- 推荐必须解释供应链传导路径：事件 → 供给/需求变量 → 价格/库存/产能 → 行业 → 公司受益环节。
- 个股结论必须区分“产业逻辑正确”和“当前估值/价格适合买入”。
- 每个观点都必须可回测，并记录当时可见数据，避免未来数据泄漏。
- 飞书推送失败必须重试并告警，但不得重复推送同一 run_id。
