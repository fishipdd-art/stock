# Dify 供应链错配投资研究系统 V1 需求

> 状态：V1 需求已确认；供后续智能体实现使用。

## 1. 产品定位

系统服务单一用户，将免费公开信息转化为可解释、可验证、可回测的供应链投资研究建议。核心价值不是“新闻摘要”，而是尽早发现供需变化、验证传导逻辑、判断市场是否已计价，并给出与用户持仓相关的行动建议。

## 2. Dify 中必须可见的工作流

采用“总控工作流＋多个子工作流”的可视化结构：总控画布展示完整阶段，六个子 Workflow 可分别展开、重跑和监控；另建一个 Chatflow 作为日常控制台，避免单个巨型工作流难以排错。

生产环境由 Dify 作为唯一主调度器。Python APScheduler 默认关闭，只保留手工灾备开关；必须通过全局调度锁确保 Dify 与 APScheduler 不能同时运行同一生产任务。

| 编号 | Dify 应用 | 触发方式 | 核心输出 |
|---|---|---|---|
| WF-01 | 数据采集与质量门禁 | 工作日 05:30/06:00/07:00 | run_id、各源数量、新鲜度、失败源、质量等级 |
| WF-02 | 新闻事件抽取与证据聚类 | WF-01 成功后 | 结构化事件、去重簇、正反证据、可信度 |
| WF-03 | 供应链错配与图谱传播 | WF-02 成功后 | 供需方向、传导路径、受益/受损环节、候选行业 |
| WF-04 | 行业与股票评分 | WF-03 成功后 | 行业排名、ETF/股票候选、评分拆解、风险过滤 |
| WF-05 | 持仓诊断与早报 | 工作日 08:05 | 持仓动作、候选机会、现金建议、08:20 飞书卡片 |
| WF-06 | 盘后复盘与回测记录 | 工作日 20:00 | 当日验证、观点变化、偏差归因、20:30 飞书摘要 |
| CF-01 | 供应链投资控制台 | 用户对话 | 查运行状态、追问证据、查持仓、手动重跑、解释评分 |

## 3. Dify 主工作流节点规范

每个工作流均使用清晰分区：定时触发 → HTTP 调用 → 参数校验 → If/Else 质量门禁 → MiniMax 结构化分析 → JSON Schema 校验 → 持久化 → 通知 → 结束。

所有 HTTP 节点必须设置超时、有限重试和幂等键。所有 LLM 节点必须输出固定 JSON，不允许下游解析自由文本。关键节点输出包含 `run_id` 并写入运行台账。

当质量门禁不通过时，进入降级分支：生成“数据不足报告”，列出缺失项，不输出买入建议；严重异常则仅推送运维告警。

## 4. 免费公开数据源

- 公司信息：巨潮资讯公告、沪深交易所公告。
- 新闻：新浪财经、东方财富、财联社公开页面或可稳定访问的公开接口。
- 行情：AKShare 可用数据源、交易所公开行情、Sina fallback。
- 商品与期货：国内期货主力合约、公开库存/仓单数据。
- 宏观：国家统计局、海关总署、人民银行、发改委、工信部。
- 行业证据：协会公开数据、企业公告中的产能/订单/价格信息。

每条证据保存 `source_name`、`url`、`published_at`、`collected_at`、正文 hash、来源等级和版权合规字段。系统不得依赖搜索结果摘要作为唯一证据。

## 5. 事件和错配模型

新闻采用两阶段分析：Python 完成正文抽取、规范化、去重、时间校验和明显噪声过滤；MiniMax M2.7 Highspeed 对全部候选新闻执行结构化事件抽取、相关性复核、正反证据分析和供应链路径推理。不得只依赖关键词高匹配结果，也不把未经预清洗的全部原始新闻直接交给 LLM。

结构化事件至少包含：实体、产品、产业环节、地区、事件类型、供需方向、影响幅度、开始/结束时间、证据列表、反向证据、置信度。

错配评分建议拆解为：证据可信度 20%、多源一致性 15%、供需强度 20%、价格/库存验证 15%、图谱传导强度 15%、时效性 10%、可交易性 5%。不得让 LLM 直接生成最终分数；LLM 提取事实，Python 使用可配置公式计算。

## 6. 行业与股票推荐

推荐卡必须包含：行业、标的、观点方向、综合分、催化窗口、供应链传导路径、三条以内关键证据、反向证据、市场是否已计价、观察/介入区间、止损参考、逻辑失效条件、仓位风险提示（非硬限制）。

硬过滤项包括：ST/退市风险、重大减持、财务异常、停牌、流动性不足、近期极端涨幅、证据过期、数据质量不合格。

对用户持仓输出 `加仓候选 / 持有观察 / 减仓 / 退出验证`，不得因为成本价自动建议补仓，也不得只因浮亏建议卖出。

## 7. 风险与组合管理

- 参考资金：50 万元。
- 资产范围：只考虑用户确认的 9 个持仓和账户现金，不纳入任何外部资产。
- 最大容忍回撤：20%，但系统应在 12%、16%、20% 设置三级预警，不能等到 20% 才行动。
- 不设置单股、ETF、产业链或单笔交易的硬性仓位上限。
- 系统可将总资产 1% 作为新增交易的参考风险预算，但只提示，不阻断。
- 高相关产业链按一个风险桶展示 ETF 和个股的重复暴露，只提供预警和情景分析。
- 每日报告展示现金比例、行业暴露、单股集中度、相关性集中度和压力情景。

当前组合基准数据：总资产 500,000 元，持仓市值约 465,033 元，现金约 34,967 元。持仓为 000878、002149、002859、159206、300004、300408、300502、301308、563380。

## 8. Dify 运行监控

在 Dify 控制台中能够看到每个 Workflow 的最近状态、开始/结束时间、数据量、质量等级、失败节点、重试次数、LLM token/成本和飞书送达状态。

Python 提供统一 `/api/v1/runs`、`/api/v1/health/data`、`/api/v1/pipeline/*` 接口；Dify 定时读取并生成监控摘要。外部 Grafana 继续保留做基础设施监控，但用户日常只需在 Dify Chatflow 中问“今天运行正常吗”。

## 9. 验收标准

- 连续 10 个交易日自动完成早报，08:20 前送达率不低于 95%。
- 任一核心源返回 0 条或数据过期时，必须触发质量异常而非成功。
- 抽样 100 条新闻，错误产业链匹配率目标低于 10%。
- 每条强推荐至少有两个独立证据源，或一个一级权威源加价格/库存验证。
- 所有推荐可追溯到原始证据和工作流 run_id。
- 飞书重复推送率为 0，失败有重试和告警。
- 回测严格使用当时可见数据，报告基准、交易成本、最大回撤和胜率。

## 10. 实施顺序

1. 修复调度与数据质量状态，补统一 Pipeline API。
2. 在 Dify 建 WF-01 和运行监控台账。
3. 建事件抽取、证据聚类和图谱传播。
4. 建持仓模型、评分和风险桶。
5. 建飞书早报、突发提醒和复盘。
6. 完成回测、10 日影子运行和阈值校准后，再启用明确的介入区间建议。

## 12. 2026-07-12 故障修复约束

- Dify 服务入口固定为 `http://localhost`（Docker Nginx 映射 80/443）；不得改为 28080/14443 而不同时更新全部入口与运行手册。
- Dify HTTP 节点不得将 HTTP 202 当作 Pipeline 成功。分析类 Pipeline 使用 `wait_for_completion=true`，并只依据最终的 `status` 与 `quality_status` 决定后续分支。
- 当 A 股行情过期、评分没有通过硬过滤的候选，必须生成“数据不足/仅观察”状态；不得生成买入建议或将 Dify Run 标为业务成功。
- 错配和股票综合分严格为 0-100，自动化测试必须覆盖分数上限。
- 替换工作流前先停用旧定时计划，避免同一工作日多套应用重复调度和重复推送。
- 本机 Dify 1.15.0 的触发器开关修复为本地 API 镜像补丁；升级 Dify 后必须重新验证“已停用/已运行”切换，避免 SQLAlchemy DetachedInstanceError 回归。
